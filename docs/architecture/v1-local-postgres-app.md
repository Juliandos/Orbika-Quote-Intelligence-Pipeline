# Orbika Local V1 Architecture

Date: 2026-06-17

Scope: Phase 1 architecture and local operating model for
`/home/julian95/projects/Orbika-Quote-Intelligence-Pipeline`.

This document does not implement PostgreSQL, does not create migrations, and
does not change the current CLI behavior. It adapts the architecture document
generated by OpenClaw in `TASK-20260617-003` into the Orbika repository.

## Confirmed Decisions

- Orbika remains separate from OpenClaw. OpenClaw coordinates tasks and
  operational memory; Orbika keeps its own repository, pipeline, console and
  local outputs.
- FastAPI already exists in `apps/api/orbika_console_api` and is the local V1
  API layer for tasks, state, quotes and SSE events.
- Next.js already exists in `apps/web` and is the local V1 frontend for the
  workshop operator.
- PostgreSQL is a future phase. V1 architecture prepares for it, but this phase
  does not create a database, migrations or persistence rewrites.
- Execution remains manual for now. The local stack can be started manually
  while the product experience continues moving toward a normal Windows app
  launcher later.
- The current CLI remains compatible. Scripts under `tools/` keep their current
  routes and expected outputs until a later migration is validated.
- Snapshots are not a default product feature. They may be enabled temporarily
  for debugging, then retained briefly and hidden from the operator.
- The final operator should not depend on folders, JSON files or terminals.

## Local V1 Architecture

```text
Gmail read-only
  -> tools/incremental_orbika_quote_runner.py
  -> Orbika quote page read-only through Playwright
  -> local/orbika_incremental/quotes/<quote_key>.json
  -> supplier matching
  -> optional agentic review
  -> FastAPI local console API
  -> Next.js local console
  -> workshop review and decision
```

### Components

- `tools/gmail_quote_extractor.py`: obtains quote links from Gmail with
  read-only scope.
- `tools/orbika_quote_extractor.py`: opens Orbika quote pages with Playwright
  and extracts visible quote data without modifying Orbika.
- `tools/incremental_orbika_quote_runner.py`: coordinates Gmail, Orbika,
  incremental state, quote outputs, snapshots and daily reports.
- `tools/supplier_quote_matcher.py`: enriches quote payloads with supplier
  candidates from local supplier catalog snapshots.
- `tools/agentic_match_reviewer.py`: adds assisted review over supplier
  matches when applicable.
- `apps/api/orbika_console_api`: local FastAPI API that reads current artifacts,
  launches existing tasks and exposes state to the frontend.
- `apps/web`: local Next.js console so the workshop operator can work without
  opening JSON files.
- `supplier_catalog/providers`: supplier catalogs and provider extraction
  artifacts.
- `local/`: local outputs ignored by Git and used as the current V1 operating
  source.

## Source Of Truth By Phase

### V1 Local

The current operational source of truth remains the local filesystem:

- `local/orbika_incremental/state.json`
- `local/orbika_incremental/quotes/*.json`
- `supplier_catalog/providers/*/provider.json`
- `supplier_catalog/providers/*/snapshots/*/extracted.json`

FastAPI and Next.js should hide these artifacts from the operator. The operator
works through the visual console; technical files remain support material.

### Future PostgreSQL Phase

PostgreSQL should become the main operational source once the visual workflow is
stable enough to run without terminal work. The migration should preserve a
temporary compatibility period so current scripts keep working while the API and
UI move to database-backed reads.

## Data Planned For PostgreSQL

These domains should move to PostgreSQL because they are queried, filtered,
related, audited or used in daily decisions:

| Domain | Data | Practical reason |
| --- | --- | --- |
| `emails` | Gmail IDs, received date, sender, subject, extraction status, quote keys | Keeps idempotency and backlog tracking without relying only on `state.json`. |
| `quotes` | `quote_key`, notice ID, masked URL, source email, status, warnings, timestamps | Enables search, dashboard state, retries, deduplication and retention. |
| `vehicles` | plate, brand, line, version, year, VIN and quote relationship | Enables compatibility checks and better supplier matching context. |
| `workshops` | commercial name, delivery workshop, city and address | Keeps repeated business context structured. |
| `parts` | extracted requested parts, quantity, references, raw status and quote relation | Supports review, matching, coverage and reporting. |
| `supplier_matches` | candidates by part, provider, product, score, type, reasons and URLs | Central data for purchase/recommendation decisions. |
| `agentic_reviews` | selected top options, confidence, brief comments, reviewer mode and trace pointer | Shows assisted reasoning without loading full traces in the UI. |
| `customer_preferences` | supplier preferences, risk tolerance, accepted brands, notes and rules | Lets future agentic review adapt to the workshop's real habits. |
| `tasks` | run kind, status, timestamps, command intent, counters and log pointer | Gives operational audit for "what happened today". |
| `events` | pipeline and UI events | Supports live activity, troubleshooting and lightweight audit. |
| `daily_summaries` | compact daily counts, pending items and coverage summary | Avoids relying on large daily JSON/Markdown files for dashboard data. |

Future agent/RAG tables should be prepared in a later phase:

- `rag_documents`
- `rag_chunks`
- `agent_sessions`
- `agent_messages`
- `business_memory`
- `human_handoffs`

Design rule: store in PostgreSQL what is searched, filtered, related, audited or
used for a user decision. Keep files only for heavy, temporary, regenerable or
debug-oriented artifacts.

## Daily Operating Flow

The target operator flow is visual and manual:

1. Open the local Orbika console in the browser.
2. Review system status: last check, new quotes, tasks in progress, errors and
   last update time.
3. Press a visual action to check for new quotes or keep the runner waiting.
4. Receive visual and sound notification when a valid quote is processed.
5. Review the quote inbox: customer, vehicle, date, requested parts, matching
   status and alerts.
6. Open a quote and review each requested part with supplier candidates.
7. Mark operational decisions: accepted candidate, pending, discarded, needs
   call, needs manual review or no sufficient offer.
8. Run supplier matching or agentic review only from UI controls for selected
   quotes or the current batch.
9. Review daily summary from the UI: received, covered, pending, no-match and
   failed quotes.
10. End the day with a clear pending list.

The operator should not need to know that `quotes/*.json`, `state.json`,
snapshots or traces exist.

## Retention Policy

| Type | Retention | Action |
| --- | --- | --- |
| Operational quotes | 90 days | Keep recent work fast and focused. Older records can be archived or compacted. |
| Task and activity logs | 30 days | Keep recent troubleshooting evidence only. |
| Debug snapshots | 7 days | Disabled by default; purge quickly when temporarily enabled. |
| Full agentic traces | 30 to 90 days depending on debugging value | Keep compact review summaries longer in DB; purge bulky traces. |
| Daily summaries | Compact long-term summaries | Keep aggregate counts and business metrics, not large regenerated files. |
| Temporary/staging files | 7 days | Purge incomplete staging when no task is active. |
| Supplier catalog snapshots | Keep active snapshot and useful history | Needed to understand which provider data produced a recommendation. |
| Technical caches | As needed | Safe to remove when no process is active. |

No secrets, OAuth tokens, passwords, browser storage state or full sensitive URLs
should be committed, documented in versioned files or stored in PostgreSQL.

## Criteria For Removing Operator Dependency On Folders And JSON

The dependency is considered reduced when:

- The console shows quote list, quote detail, parts, candidates, warnings and
  status without opening local files.
- Daily actions exist as UI controls: check new quotes, run matching, review
  selected quotes, retry failures and view daily summary.
- Errors show operational next steps instead of stack traces or file paths.
- The UI filters by date, status, provider, coverage and pending work.
- Quote status is clear: processed, skipped, pending, failed or needs retry.
- The operator can confirm or discard supplier candidates from the UI.
- The daily summary is visible and exportable without opening `daily/*.md` or
  `daily/*.json`.
- `local/orbika_incremental/*` appears only in technical support screens or
  logs, not in the normal workshop workflow.
- The CLI remains available for technical support with current defaults.

## Non-Goals For This Phase

- Implement PostgreSQL.
- Create models, migrations or tables.
- Rewrite CLI scripts to use a database.
- Change FastAPI endpoints.
- Change Next.js components.
- Automate purchasing or external actions.
- Execute Gmail, Orbika or supplier actions while documenting architecture.
